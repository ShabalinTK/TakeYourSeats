Attendance Control System (ACS) — Web Application
Welcome
“Everything in software architecture is a trade-off. First Law of Software Architecture”

ACS — это веб-система для контроля посещаемости, использующая интерактивные карты рассадки и динамические QR-коды для верификации присутствия студентов в аудитории.

Business Case
Текущие методы контроля посещаемости в ВУЗах страдают от недостоверности (пересылка статичных кодов) и временных затрат на перекличку.

Бизнес-цели:

Достоверность: Исключение возможности регистрации студентов вне аудитории.

Прозрачность: Визуальная сверка карты приложения с реальной рассадкой студентов.

Эффективность: Автоматизация формирования отчетности для преподавателей и администрации.

System Requirements
Stakeholders
SH-1: Администратор/Преподаватель: управляет учетными записями, создает сессии занятий и контролирует отчетность.

SH-2: Студент: осуществляет регистрацию на занятии и бронирование физического места.

SH-3: Менеджмент (Деканат): отслеживает операции с тикетами (посещаемостью) и генерирует финансовые/учебные отчеты.

Functional Requirements
UC-1: Управление аккаунтами: администратор поддерживает учетные записи преподавателей и данные студентов.

UC-2: Ticket Workflow (Регистрация):

Преподаватель запускает сессию и транслирует динамический QR-код.

Студент сканирует код, выбирает место на карте и подтверждает выбор.

UC-3: Мониторинг: преподаватель в реальном времени видит заполнение аудитории на схеме.

UC-4: Отчетность: генерация отчетов по посещаемости в форматах Excel/CSV.

UC-5: Нотификация: (опционально) отправка уведомлений студентам о начале регистрации.

Architecture Characteristics Requirements
QA-1: Security (Безопасность):

Dynamic QR: использование временных токенов в ссылках для предотвращения передачи доступа третьим лицам.

Session Isolation: каждая сессия уникальна для конкретной аудитории и времени.

QA-2: Performance (Производительность): время отклика системы при выборе места должно быть минимальным (< 1-2с) для предотвращения задержек на входе в аудиторию.

QA-3: Usability (Удобство): адаптивный интерфейс (Mobile-first) с механизмом «выбор → подтверждение» для исключения случайных нажатий.

Constraints
CON-1: Система должна быть доступна через веб-браузер без установки дополнительного ПО.

CON-2: Интеграция с базой данных планировок аудиторий.

Assumptions
ASM-1: Мобильное приложение (в данном случае веб-интерфейс) является частью системы и может быть изменено.

ASM-2: Студенты имеют доступ к личному смартфону с выходом в интернет.

ASM-3: Преподаватель имеет техническую возможность продемонстрировать QR-код группе.
